# plan_0007 - 서비스 폴더 재배치 및 경로 최신화

## 상태
- 완료 (2026-02-07)

## 목표
- 서비스 서브모듈 디렉토리를 `services/`로 정리하고,
  테스터 코드를 `tester/src/tester`로 이동해 런타임/문서 경로를 일관되게 최신화한다.

## 전제
- 실제 이동 대상과 목적 경로는 사전 확정 필요.
- CI/CD, compose, 로컬 스크립트, 테스트가 참조하는 상대 경로를 모두 추적해야 함.

## 작업 단위
1. 경로 영향도 인벤토리
- 대상: `.gitmodules`, `docker-compose*.yml`, `bin/*`, 각 서비스 `docker-compose*.yml`, 문서, 테스트 스크립트
- 조치: 하드코딩 경로 목록 추출 및 변경 매트릭스 작성

2. Git submodule 경로 변경
- 조치:
  - `.gitmodules`의 `path` 갱신
  - 인덱스/워크트리 기준 submodule 경로 이동 반영
  - 필요 시 submodule sync 절차 문서화

3. 런타임 경로 최신화
- 대상: `docker-compose.local.yml`, 서비스 Docker build context, 볼륨 경로, 스크립트 경로
- 조치: 새 디렉토리 구조 기준으로 모두 갱신

4. 코드/문서 참조 경로 최신화
- 대상: `docs/*`, `README/HANDHELD`, 테스트/유틸 스크립트, 테스터 import 경로
- 조치: 실행 커맨드와 경로 예시를 최신 구조로 정렬

5. 검증
- `./bin/project config` 성공
- `./bin/project up` 기동 확인
- 최소 헬스체크 + 스모크 테스트 통과

## 리스크
- submodule path 변경 시 로컬 개발환경 캐시/참조 꼬임 발생 가능
- compose build context 오타가 전체 기동 실패로 연결될 수 있음

## 완료 기준 (DoD)
- 새 폴더 구조에서 submodule 초기화/업데이트/기동이 모두 재현 가능.
- 경로 불일치로 인한 compose/스크립트 실패가 없어야 함.
- 테스터 패키지(`tester/src/tester`) import 및 모듈 실행 경로가 정상 동작해야 함.

## 검증(테스트) 결과
1. Submodule 경로 검증
- 명령: `git submodule status`
- 결과: 모든 항목이 `services/*` 경로로 출력됨.

2. Compose 구성 검증
- 명령: `./bin/project config`
- 결과: 성공 (`OK`), 경고는 API Key 미설정 관련 only.

3. 서비스 목록 검증
- 명령: `docker compose -f docker-compose.local.yml -p gtrpg-local config --services`
- 결과: 9개 서비스(`postgres`, `redis`, `be-router`, `gm`, `state-manager`, `scenario-service`, `rule-engine`, `llm-gateway`, `web`) 정상 출력.

4. 통합 기동 스모크
- 명령: `./bin/project up`, `./bin/project ps`
- 결과: 경로 변경으로 인한 build context 오류 없이 기동.
- 관찰 이슈: `llm-gateway`는 `GOOGLE_API_KEY` 미설정으로 재시작(경로 이슈 아님, 기존 환경 변수 이슈).

5. 환경 정리
- 명령: `./bin/project down`
- 결과: 생성된 컨테이너/네트워크 정상 정리.

6. 테스터 모듈 경로 검증
- 명령: `PYTHONPATH=tester/src uv run python -c "from tester.runner import IntegrationTestRunner; from tester.agent import TesterAgent; print('ok')"`
- 결과: import 성공(`ok` 출력).
