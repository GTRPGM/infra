# plan_0012 - BE-router 전체 게임 플로우 정상 동작 검증

## 상태
- in_progress (시작일: 2026-02-07)

## 목표
- BE-router를 통과하는 핵심 사용자 플로우가 실제로 정상 동작함을 검증한다.
- 검증 축을 다음 2가지로 고정한다.
  1. 시스템 정상 동작: 각 엔드포인트가 의도한 목적에 맞게 기능하고, 조회 응답이 다음 단계 진행에 충분한 정보를 제공해야 한다.
  2. 턴/상태 정합성: 턴 진행 결과가 상태 변화로 정확히 반영되어야 하며, 불가능한 행동은 거절되어야 한다.

## 핵심 사용자 플로우(검증 대상)
1. 로그인
2. 시나리오 선택
3. 세션 생성
4. 세계 상태 요약 조회
5. 턴 진행(복수 턴)

## 작업 단위
1. 계약/경로 동결 (BE-router 기준)
- BE-router 공개 OpenAPI를 기준으로 실제 엔드포인트 경로/필수 필드/응답 필드를 고정한다.
- 대상 문서:
  - `docs/dev/detail/ref_contract_matrix.md` (필요 시 섹션 추가)
- 산출물:
  - 플로우 단계별 호출 엔드포인트 표
  - 단계 간 필수 전달 필드 표(`user_id`, `scenario_id`, `session_id`, `act_id`, `seq_id`, `turn_id` 등)

2. 시스템 정상 동작 검증 시나리오 작성
- `tester/src/tester`에 "BE-router 단일 진입" E2E 시나리오를 추가한다.
- 각 단계에서 아래를 검증한다.
  - 상태코드/응답 구조가 계약과 일치
  - 다음 단계 수행에 필요한 식별자/상태 정보가 응답에 존재
  - 실패 시 실패 지점(단계/엔드포인트/누락 필드)을 즉시 진단 가능

3. 턴/상태 정합성 검증 시나리오 작성
- 정상 시나리오:
  - 이동/상호작용/획득 등의 행동 수행 후 상태 조회로 반영 여부 검증
- 부정 시나리오:
  - 존재하지 않는 아이템 획득 시도 -> 실패 또는 거절 메시지 확인
  - 현재 맥락에 없는 장소 이동 시도 -> 실패 또는 거절 메시지 확인
  - 실패 시 인벤토리/관계/위치 상태가 오염되지 않았는지 확인

4. 관측 및 증빙 표준화
- 턴별 증빙 아티팩트를 고정한다.
  - 요청/응답 요약
  - 턴 전/후 상태 스냅샷 diff
  - commit/turn progression 추적 정보
- 실패 유형을 `contract`, `state_sync`, `transition`, `narrative_only`로 분류한다.

5. 회귀 게이트 연결
- 단기: 5턴 시나리오(`smoke`)
- 중기: 10턴 시나리오(`smoke10`)
- 장기: 20턴 시나리오(`endurance20`)
- 각 게이트에서 "정상/부정 시나리오"를 최소 1개씩 통과해야 done으로 승격한다.

## 검증 설계 (두 평가축 매핑)
- 축 1. 시스템 정상 동작
  - 로그인->시나리오 선택->세션 생성->상태요약->턴 호출까지 단계별 성공 및 필드 충분성 검증
  - 조회 API(`session`, `sequence/details`, `act/details`, `inventory`)가 판단 근거로 충분한지 검증
- 축 2. 턴/상태 정합성
  - action 결과가 state commit 이후 조회값과 일치하는지 검증
  - 불가능 행동 거절 시 상태 불변성(인벤토리/위치/관계) 검증
  - 시퀀스/액트 전이가 발생했을 때 canonical ID 규약(`act-*`, `seq-*`) 유지 검증
