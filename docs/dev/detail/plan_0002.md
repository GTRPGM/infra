# plan_0002 - Scenario Service -> State 스키마 완전 정렬

## 목표
- Scenario 주입 payload가 State 주입 계약과 **완전 호환**되도록 정렬한다.
- 단, State 내부 런타임 ID(`npc_id`, `enemy_id`, `item_id` UUID 등) 생성 책임은 State에 유지한다.

## 핵심 원칙
- 시나리오 엔티티/관계 스키마는 State와 동등해야 한다.
- 허용 차이: State 내부 인스턴스 식별자(내부 PK/UUID)만 제외.

## 작업 단위
1. DTO 정렬
- 대상: `services/scenario-service/src/scenario/core/models/generation.py`
- 조치:
  - `master_id` -> `rule_id` 변환 규칙 명시
  - `item_id`/`scenario_item_id` 규약 통일
  - `relations.from_id/to_id` 타입/값 규약을 State와 일치

2. 패키징 로직 정렬
- 대상: `services/scenario-service/src/scenario/core/engine/scenario_engine.py`
- 조치:
  - `_package_scenario`가 State `ScenarioInjectRequest` 호환 payload 생성
  - acts/sequences/npcs/enemies/items/relation 참조 무결성 검증 추가

3. 주입 어댑터 정렬
- 대상: `services/scenario-service/src/scenario/plugins/db/adapter.py`
- 조치:
  - inject 직전 payload 검증(필수 필드/타입/참조) 수행
  - State로 넘기는 필드만 유지(확장 메타는 내부 저장으로 분리)

4. 계약 테스트 추가
- 대상: `services/scenario-service/tests/*`
- 조치:
  - State schema mirror 검증 테스트
  - 참조 ID 무결성 테스트(acts->seq, seq->entities, relations)

## 리스크
- 기존 생성 결과의 ID 정규화 규칙 충돌 가능
- 드롭아이템/관계의 식별자 체계 혼재 가능

## 완료 기준 (DoD)
- `POST /api/v1/generation/pure` -> `POST /api/v1/manage/scenarios/{id}/inject` 연쇄에서 schema mismatch 0건.
- State 주입 시 422/500 없이 성공하고, 조회 결과에서 참조 일관성 보장.

## 실행 결과 (2026-02-07)
- DTO 정렬 완료 (`scenario-service` 내부 canonical 기준):
  - `item_id/master_id` 중심 구조를 `scenario_item_id/rule_id` 중심으로 정렬.
  - 하위 호환을 위해 legacy 키(`item_id`, `master_id`) 입력은 계속 수용.
- 패키징 로직 정렬:
  - `_package_scenario`가 캐노니컬 필드(`scenario_item_id`, `rule_id`) 생성.
  - `_to_state_injection_payload` 직전 참조 무결성 검증(`acts->sequences`, `sequences->entities`, `relations`) 추가.
- DB 어댑터 정렬:
  - `save_scenario` 진입 전에 참조 무결성 검증 추가.
  - 아이템 식별자 처리에서 `scenario_item_id` 우선 사용, `item_id` fallback 유지.
