## 문제
- 현재 GM 파이프라인/NPC 턴 결과는 주로 단일 `narrative` 문자열에 섞여 내려오며,
  - "NPC가 말한 대사"와 "NPC/적이 수행한 행동"을 클라이언트(또는 WEB)에서 안정적으로 분리해 표현하기 어렵다.
  - 추후 리플레이/자막/로그 UI에서 "말"과 "행동"을 다른 스타일로 렌더링하기 어려움.

## 목표
- NPC/적 턴에서 "대사(dialogue)"를 **응답에 별도 필드로 제공**하여 클라이언트가 안정적으로 자막/말풍선을 렌더링할 수 있게 한다.
- "행동(action)"은 기본적으로 서사(`narrative`)에 녹여서 표현하되,
  - **무대사 행동도 UI가 표현할 수 있도록** `action`을 응답에 포함한다.
  - 클라이언트는 `dialogue`가 비어있는 경우 `action`을 보조 UI로 표시할 수 있다.

## 범위

  - `BE-router` 응답 패스스루(필요 시)
  - `tester`가 구조화 출력이 있을 때 이를 기록/검증하는 보강(선택)

## 제안 계약 (Draft)
- GM 턴 응답(`GameTurnResponse`)에 다음 필드를 추가한다.
  - `dialogue`: string | null
    - NPC/적의 직접 발화. 발화가 없으면 null.
    - `narrative`와 별개로 클라이언트가 그대로 표시 가능해야 한다.
- GM 턴 응답(`GameTurnResponse`)에 다음 필드를 추가한다.
  - `action`: string | null
    - NPC/적의 행동 원문(대사 제외). 발화가 없더라도 행동을 UI에서 표시할 수 있도록 제공한다.
- GM 턴 응답(`GameTurnResponse`)에 다음 필드를 추가한다.
  - `outputs`: array
    - 나레이션/대사 출력 조각 리스트.
    - 각 원소는 최소 다음을 포함:
      - `kind`: `"narration"` | `"dialogue"`
      - `text`: string
      - `actor_type`: `"narrator"` | `"npc"` | `"enemy"` | `"player"` | `"unknown"`
      - `actor_id`: string | null
      - `actor_name`: string | null
- 기존 `narrative`는 하위 호환을 위해 유지한다.
- **action은 응답에 포함하지 않는다.**

## 구현 메모 (Idea)
- NPC/적 턴에서 LLM 출력 스키마를 `action`/`dialogue`로 구조화해서 받는다.
  - `action`은 내부 `TurnContext.user_input`으로만 흘려서 룰엔진/나레이션 생성에 사용
  - `dialogue`는 내부 `TurnContext.npc_dialogue`에 저장 후 API 응답에만 노출

## 완료 기준 (DoD)
- NPC/적 턴이 발생하는 경우, GM API 응답에 `dialogue`가 존재(발화 시)하거나 null(발화 없음)로 안정적으로 내려온다.
- `dialogue`와 `narrative`의 역할이 분리된다.
  - `dialogue`: 직접 발화만
  - `narrative`: 행동 결과를 포함하는 서술(행동 원문 자체를 별도 필드로 노출하지 않음)
- 기존 소비자(단일 `narrative`)도 깨지지 않는다.

## 구현 노트
- `gm`의 NPC/적 턴 입력 생성(`generate_npc_input`)은 `{action, dialogue}` JSON을 생성하도록 유도하고,
  GM이 이를 파싱하여:
  - `action` -> 내부 `user_input`으로만 흘려 룰엔진/상태/서사 입력으로 사용
  - `dialogue` -> `TurnContext.npc_dialogue`로 저장 후 응답에만 노출
- 응답에는 `action`도 포함한다(무대사 행동 표시 목적).
- NPC/적 턴의 주체(actor) 선택은 LLM 선택을 제거하고, `sequence_type=COMBAT`에서는 생존 적을 우선 선택하도록
  결정적(deterministic) 선택으로 변경했다.
