# plan_0008 - DB 분리 및 마이그레이션 (Graph/Play/Rule)

## 상태
- drifted (재정렬 필요, 점검일: 2026-02-07)
- 근거: `docker-compose.local.yml`이 현재 단일 `postgres` 서비스를 사용하며, 플랜의 "분리 구동 완료" 상태와 불일치.

## User Requirements
- 기존 단일 DB 구조를 목적별로 분리하여 확장성과 관리 효율성을 높여야 함.
- **타겟 DB 구조**:
  1. **Graph DB (`state_db`, `scenario_db`)**: Apache AGE 기반. 상태 및 시나리오 그래프 데이터 저장.
  2. **Play DB (`play_db`)**: `pgvector` 등 활용 예정. 플레이 로그 및 RAG를 위한 벡터 데이터 저장.
  3. **Rule DB (`rule_db`)**: 룰 관련 정적(Read-only) 데이터 및 유저-세션 매핑 관계 저장.
- **마이그레이션**:
  - `db/migration/` 내의 덤프 및 필터링 스크립트(`filter_dump.py`)를 활용하여 기존 데이터 이관.
  - 테이블별 적절한 DB로의 재배치.
- **스키마 정의**: 추후 사용자가 상세 테이블 매핑 정보를 제공할 예정.

## Goal
- 3개의 독립된 DB 인스턴스(또는 논리적 DB)로 인프라 재구성.
- 기존 데이터를 손실 없이 각 목적에 맞는 DB로 마이그레이션.
- 각 서비스(`gm`, `state-manager`, `rule-engine` 등)가 올바른 DB를 바라보도록 설정 변경.

## Scope
- **In**:
  - `docker-compose` 인프라 설정 변경 (DB 컨테이너 분리 또는 DB 생성).
  - `db/migration` 스크립트 고도화 (덤프 분할 및 로드).
  - 서비스별 DB 연결 설정(`configs`) 업데이트.
- **Out**:
  - 애플리케이션 비즈니스 로직의 대규모 리팩토링 (DB 연결 부만 수정).

## Implementation Plan
- **Step 1: 인프라 구성 (Docker Compose)**
  - `docker-compose.local.yml` 및 `services/*/docker-compose.yml` 수정.
  - Postgres 서비스 분리 (e.g., `postgres-graph`, `postgres-play`, `postgres-rule`) 또는 단일 클러스터 내 DB 분리 확인.
- **Step 2: 마이그레이션 스크립트 작성**
  - `filter_dump.py`를 확장하여 각 DB별 전용 덤프 파일 생성 로직 구현.
  - `run_migration.sh` 작성: DB 초기화 -> 덤프 주입.
- **Step 3: 스키마 및 데이터 이관**
  - 사용자로부터 테이블 매핑 정보 수령 후 적용.
  - 데이터 주입 테스트.
- **Step 4: 서비스 연결 설정 변경**
  - 각 서비스의 환경변수 및 `configs/db.py` 수정.

## Unit Test Plan (must)
- **What to test**: 마이그레이션 스크립트의 필터링 로직 정확성.
- **Test cases**:
  - `scenario` 테이블이 Graph DB 덤프에 포함되는가?
  - `play_logs` 테이블이 Play DB 덤프에 포함되는가?
  - `filter_dump.py` 실행 후 생성된 SQL 파일의 무결성 검사.
- **Command**:
  - `cd db/migration && python -m pytest tests/` (테스트 스크립트 신설 필요 시)

## E2E Plan (must)
- **Scenario**:
  1. 로컬 환경에서 기존 DB 덤프 준비.
  2. 마이그레이션 스크립트 실행.
  3. `bin/project up`으로 새로운 DB 환경 구동.
  4. 각 DB에 접속하여 테이블 및 데이터 존재 확인.
  5. GM 서비스 등에서 정상적으로 데이터 조회 가능한지 확인.
- **Expected**:
  - `state-manager`는 Graph DB에서 상태 조회 성공.
  - `gm`은 Play DB에 로그 적재 성공.
- **How to run**:
  - `./db/migration/run_migration_test.sh` (가칭)
- **Validation**:
  - `psql` 접속 후 `\dt`로 테이블 목록 확인.

- [ ] 애플리케이션이 변경된 DB 접속 정보를 통해 정상 기동됨.
