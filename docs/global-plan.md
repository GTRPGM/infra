# TRPG LLM GM Core 설계서

## 0. 목적과 범위

이 문서는 LLM 기반 TRPG 진행 시스템의 **코어 서비스 설계**를 정의한다. 목표는 다음을 보장하는 것이다.

- 자연어 입력(플레이어/운영자/NPC 행동)을 **구조화된 판정 파이프라인**으로 처리한다.
- 규칙(룰)과 시나리오 제약을 기반으로 결과를 산출하되, 최종 확정은 **GM**이 수행한다.
- 세계 상태 변경은 **단일 커밋**으로만 발생하며, **커밋 성공 이후에만 서술**이 생성된다.
- 서비스 경계(권한/책임/DB 소유)가 명확하고, 각 파트 담당자가 이 문서만으로 인터페이스/동작을 오해 없이 구현할 수 있어야 한다.

> 본 문서는 **세부 구현(프레임워크/DB 스키마/큐/프로토콜 상세)**을 다루지 않는다. 책임/경계/정책/흐름을 고정하는 것이 목적이다.

---

## 1. 시스템 원칙

### 1.1 MSA 및 데이터 접근 원칙

- 코어는 MSA 전제를 따른다.
- 각 서비스(매니저)는 **자기 DB에만** 접근한다.
- 서비스는 기본적으로 **무상태(stateless)**를 전제로 한다. 지속 상태는 각 서비스 DB가 담당한다.

### 1.2 직렬화(동시성) 원칙

- **세션당 단일 in-flight**를 Backend Router에서 보장한다.
- 동일 세션에서 동시 입력은 허용하지 않는다(직렬 처리).

### 1.3 커밋-서술 불변조건

- 결과 처리는 반드시 아래 순서를 따른다.
  1. GM이 최종 결과/상태 변경을 확정한다.
  2. State Manager에 **단 1회 커밋**을 요청한다.
  3. **커밋 성공 후에만** 자연어 서술을 생성한다.

- 커밋 실패 시: GM은 **재시도**한다.
- 서술 생성이 요구 조건(서술 슬롯)을 만족하지 못할 경우: GM은 **재시도**한다.

## 2. 구성 요소(레이어/서비스)

### 2.1 외부 레이어

#### Backend Router

- 인증/세션 관리
- 세션당 단일 in-flight 보장
- 코어 서비스 호출 라우팅

### 2.2 코어 서비스

#### GM (Game Master Orchestrator)

##### **역할**

- 입력을 수신하고 전체 파이프라인을 오케스트레이션한다.
- Rule/Scenario 제안을 취합하여 **최종 결정**을 내린다.
- State Manager에 상태 변경 diff를 전달해 커밋을 수행한다.
- 커밋 성공 후 LLM Gateway를 통해 서술을 생성한다.
- NPC 턴의 경우 컨텍스트를 수집해 NPC 행동을 생성하고 동일 파이프라인을 반복한다.

##### **제약(정책)**

- Scenario Manager 제약은 **강제(accept-only)**: GM은 시나리오 제안을 반드시 반영해야 하며 강행 커밋이 불가하다.
- 사용자 입력 1회에 대해 **최대 1회 커밋**만 허용한다.
- 파생 판정(반동 등)이 필요하면 최대 1회 추가 판정 후 최종 결과를 확정하고 단일 커밋으로 종료한다.

##### **ID 발급/로그**

- `turn_id` 발급 주체: **GM**
- GM은 GM 전용 PlayLog DB에서 **마지막 엔트리를 읽어 다음 turn_id**를 생성한다.

#### Rule Manager

##### **역할**

- 사용자 입력을 룰/카테고리에 매핑하고 판정을 수행한다.
- 판정에 필요한 상태 조각을 State Manager에 요청한다.
- GM에 **구조화된 제안 객체**(판정 제안)를 반환한다.

##### **데이터**

- 룰 정의에는 “판정에 필요한 객체(IDs/선택 기준)”를 포함한다.

#### Scenario Manager

##### **역할**

- 시나리오 템플릿 기반으로 세션의 진행 위치/배경/NPC/사물을 관리한다.
- 현재 상태/룰 변화가 시나리오와 충돌하는지 판정하고 **제안 객체**로 반환한다.
- 충돌(불일치) 시:
  - `불일치 항목 목록`과 `항목별 강도`를 제공한다.
  - 가능한 경우 `대체/완화` 제안을 제공한다.
  - 필요 시 **서술용 슬롯**을 제공/요구한다.

- 시나리오 진행 위치 전환은 **GM이 명시적으로 요청**하여 수행한다.

##### **시나리오 불변성**

- 시나리오 자체는 세션 시작 시 1회 생성되며, 세션 중 **불변**으로 간주한다(현 단계).

#### State Manager

##### **역할**

- 세계 상태의 단일 진실 원천(Source of Truth)이다.
- 상태는 **엔티티 그래프**로 관리한다.
  - 엔티티: 최소 단위
  - 관계: 엣지

- 엔티티의 정형 값은 테이블 형태로 제공한다(엔티티 스펙은 RDB에 정의).
- GM이 전달한 상태 diff를 **단일 커밋**으로 적용한다.

##### **ID 발급**

- `commit_id` 발급 주체: **State Manager**

#### LLM Gateway

- 시스템 내 모든 LLM 호출의 단일 통로
- 사용처: 입력 해석/룰 매핑, NPC 행동 생성, 결과 서술

## 3. 핵심 데이터와 식별자

### 3.1 식별자

- `turn_id`: GM 발급. 세션 내 단조 증가.
- `commit_id`: State Manager 발급. 상태 버전 식별.
- 주요 키: `(turn_id, commit_id)` 조합.

### 3.2 엔티티 ID 규약

- 엔티티 ID는 **시나리오 템플릿에서 유일**하다.
- 동일 템플릿 엔티티는 **세션이 다르면 여러 인스턴스로 존재**할 수 있다.
- 세션 초기화 시: 시나리오 템플릿 엔티티 ID를 상태 초기 엔티티로 **복사/동일 사용**한다(전역 규약).

### 3.3 Play Log (리플레이 기준)

- 소유: **GM 전용 DB**
- 최소 이벤트 타입(필수 저장):
  - `user_input`
  - `gm_final_output`
  - `state_diff`

- 버전 메타(필수):
  - `scenario_template_id` + `scenario_version`
  - `rule_set_version`
  - `llm_model/version` (서술 및 관련 호출에 대한 일관성 목적)

- 리플레이 기준: **커밋 결과 + 그 기반 서술**

### 3.4 관측/운영 로그

- 컨테이너 stdout 전역 로그: 운영 개선/추적 목적
- LangSmith 로그: 품질 개선 및 LLM 호출 추적 목적
- 장기적으로 RAG 대상 로그는 분리/확장 가능(범위 외)

---

## 4. 제안/가드 정책(우선순위와 봉투)

### 4.1 제안 객체 원칙

- Rule Manager와 Scenario Manager의 출력은 모두 **구조화된 제안 객체**다.
- GM은 제안 객체를 취합하여 최종 결과/상태 diff를 확정한다.

### 4.2 우선순위

- **Scenario > Rule**
  - Scenario 제약은 강제(accept-only).
  - GM은 시나리오 제안을 반드시 반영해야 한다.

### 4.3 Rule 봉투(envelope)

- Scenario가 수치 결과를 조정할 때는 **Rule이 허용하는 수치 범위(봉투) 안**에서만 가능하다.
- 범주형 결과(성공/실패 등)는 봉투가 없다.
  - 시나리오가 막아야 하면 즉시 실패/거부로 처리 가능.

### 4.4 봉투 내 불가능 상황

- Rule 봉투 내에서 시나리오 목적 달성이 불가능할 경우:
  - **시나리오 목적 기준으로 보정값**을 선택한다(단순 최솟값 고정 아님).
  - Scenario Manager는 적절한 정당화를 위해 **서술용 슬롯**을 제공/요구한다.

---

## 5. 파생 판정(반동 등) 정책

- 사용자 입력 1회에 대해:
  - 파생 판정은 **최대 1회** 허용한다.
  - 파생 판정은 새 행동이 아니라 **기존 행동의 연장**이다.
  - 최종 결과는 **원행동 실패/무효 + 반동 효과**를 함께 반영할 수 있다.

- 최종적으로는 항상 **단일 커밋**으로 종료한다.

---

## 6. 시나리오 진행(위치) 정책

- 시나리오 진행 위치 전환은 **GM이 명시적으로 요청**한다.
- 전투/탐색 등의 시퀀스 종료 판정은 GM이 수행한다.
- 일반적으로:
  - 상태 변화 커밋이 선행되고,
  - 그 후 진행 위치 전환을 수행한다.

- 진행 위치 전환은 **느슨한 일관성**을 갖는다(원자적으로 상태 커밋과 묶지 않는다).

## 7. 주요 플로우

### 7.1 플레이어 턴

1. 입력 -> GM
2. GM -> Rule Manager (룰 판정 제안 요청)
3. Rule Manager -> State Manager (룰이 요구하는 상태 조각 요청)
4. Rule Manager -> GM (제안 객체 반환)
5. GM -> Scenario Manager (시나리오 정합성/영향 제안 요청)
6. Scenario Manager -> GM (불일치/항목별 강도/서술 슬롯 포함 제안 반환)
7. GM 최종 확정(accept-only 가드 준수) 및 diff 작성/수정
8. GM -> State Manager (diff 커밋 요청)
9. State Manager -> GM (commit_id 포함 커밋 결과)
10. GM -> LLM Gateway (커밋 성공 이후 서술 생성)
11. GM 최종 출력 반환 및 Play Log 기록

### 7.2 NPC 턴

1. GM -> State Manager + Scenario Manager (NPC 컨텍스트 요청)
2. GM 컨텍스트 조합 -> LLM Gateway (NPC 행동 생성)
3. 생성된 NPC 행동을 입력으로 간주하여 7.1 흐름을 반복

---

## 8. 책임/경계 체크리스트

### 8.1 변경 권한

- 상태 변경(write): **State Manager만** 수행한다.
- 상태 변경 트리거/요청: **GM만** 수행한다.
- Scenario 제약 무시/강행: **불가**

### 8.2 데이터 접근

- 각 서비스는 자기 DB만 접근한다.
- GM은 GM PlayLog DB만 소유/접근한다.

### 8.3 장애/재시도

- 커밋 실패: GM 재시도
- 서술 슬롯 불만족: GM 재시도

---

## 9. 미정(후속 설계 대상으로 남기는 항목)

아래 항목은 정책/경계에는 영향이 없으나, 구현 시 별도 설계 문서로 고정한다.

- 제안 객체 스키마(필수 필드)
- 상태 조각 요청 규격(룰이 요구하는 객체 지정 방식)
- LLM 호출 시 추적 포인터(예: trace/run ID)를 Play Log에 저장하는 방식
- 시나리오 불일치 강도 스케일의 표준화(레벨 정의)
